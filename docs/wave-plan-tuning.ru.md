# Подбор параметров wave-плана без агентов

`migration tune-wave-plan` — детерминированный эксперимент только над планом. Команда сканирует Selenium-тесты, перебирает множество профилей упаковки и формирует ранжированную рекомендацию, не запуская supervisor, executor, reviewer, watchdog, sentinel или саму миграцию.

```bash
selenium-pw-migrator migration tune-wave-plan \
  --input ./SeleniumTests \
  --workspace migration \
  --out migration/plan-tuning
```

`migration plan --wave-profile auto` выполняет тот же эксперимент автоматически перед созданием рабочего плана:

```bash
selenium-pw-migrator migration plan \
  --input ./SeleniumTests \
  --strategy wavefront \
  --workspace migration \
  --out migration/plan \
  --wave-profile auto
```

Эксперимент строит диапазоны параметров из самого inventory: количества тестов и файлов, квантилей сложности/действий и распределения тестов по source-файлам. Он меняет вместимость по тестам и файлам, мягкие и жёсткие бюджеты действий/сложности, а также предельную стоимость следующих тестов из уже исследованного файла. Кандидаты оцениваются по статической модели полной стоимости:

- оценочная работа миграции;
- количество полных циклов агентских ролей;
- координационный риск чрезмерно крупной волны;
- одиночные волны, кроме smoke;
- размазывание одного source-файла по разным волнам;
- превышения мягких бюджетов;
- неравномерность нагрузки;
- тяжёлые одиночные тесты.

Первый тест из файла оплачивает полную оценочную сложность. Следующие тесты того же файла учитываются лишь с заданным процентом, потому что контекст файла, fixtures, Page Objects, helper-семантика и многие mappings уже исследованы. Именно полное повторное сложение сложности каждого теста ранее породило почти одну волну на тест.

Артефакты:

- `wave-tuning.json` — машинный результат эксперимента и лучшие кандидаты;
- `wave-tuning.md` — рекомендация для оператора;
- `recommended-preview/waves.json` и `plan.md` — предварительный план с рекомендованным профилем.

Для любого набора тестов формируется адаптивное справочное число волн, но оно не навязывается оптимизатору. Побеждает профиль с минимальной оценочной полной стоимостью. Для маленького inventory это могут быть 2–3 волны, для среднего — единицы или низкие десятки, для большого — столько, сколько требуется безопасной вместимостью. Явный `--target-waves` превращает желаемое число волн в дополнительное ограничение модели.

Мягкие бюджеты направляют упаковку. Широкие жёсткие границы вычисляются из квантилей текущего inventory и только они дают `BLOCKED`. Статусы `PASS`, `SOFT_LIMIT_EXCEEDED` и `HEAVY_SINGLE_TEST` разрешают выполнение. В `wave-tuning.md/json` также записываются составляющие стоимости, разрыв с кандидатом №2 и уровень уверенности рекомендации.

Доступные профили:

- `auto` — рекомендуемый, подбирает параметры по текущему inventory;
- `compact` — меньше более крупных волн;
- `balanced` — универсальная упаковка;
- `conservative` — меньшие волны;
- `manual` — использует явные `--max-wave-*`, `--hard-wave-*` и `--same-file-marginal-cost`.

Для калибровки статической модели по реальному времени агентов можно передать `--role-overhead <weight>`. Чем дороже запуск ролей, ревью и final gate, тем сильнее tuner предпочитает меньшее количество волн. Это не обещание точного wall-clock времени, а воспроизводимая модель выбора компромисса.
