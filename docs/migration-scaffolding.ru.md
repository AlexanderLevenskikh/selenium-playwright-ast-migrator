# Migration scaffolding

Migration scaffolding — это ограниченный структурный fallback для проектных helper-ов и границ Page Object, которые слишком дорого исследовать внутри массовой миграции тестов. Это не suppression и не имитация рабочего поведения.

## Продуктовая цель

Мигратор должен избавить пользователя от рутины переписывания десятков и сотен тестов, а не автономно восстановить каждую зависимость проекта. Типовые действия Playwright, assertions, waits, простые POM-методы и дешёвые детерминированные побочные эффекты по-прежнему мигрируются. Редкий helper, на котором процесс застрял после одной ограниченной попытки, можно вынести в отдельную работу.

## Протокол

```text
неизвестный helper/POM-корень
→ одна ограниченная попытка обычной реализации
→ CLI вычисляет COMPLETED или NO_PROGRESS
→ COMPLETED: остаётся настоящая реализация
→ NO_PROGRESS: manager может выбрать SCAFFOLD_CURRENT_ROOT
→ добавляется одно узкое явное scaffold-правило
→ повторная генерация и раздельная оценка structural/runtime readiness
```

Конфигурация:

```json
{
  "ScaffoldMethods": [
    "TariffSettingsHelper.FindTariff"
  ],
  "ScaffoldMethodPatterns": [
    "TariffSettingsHelper.Select*"
  ]
}
```

Точное имя без receiver применяется только к неквалифицированному вызову. Для квалифицированного вызова нужно полное имя или шаблон владельца. В шаблоне владелец всегда точный, wildcard допустим только в последнем сегменте метода.

## Что генерируется

Renderer заменяет выбранный вызов на явный `__MigratorScaffoldRuntime`, сохраняет присваивание результата и `await`, добавляет `[MIGRATOR:SCAFFOLD]` и бросает `NotImplementedException` при запуске. Заглушка никогда не возвращает правдоподобное значение, из-за которого тест мог бы случайно пройти.

## Ограничители

- remediation считается `COMPLETED` только когда из generated blocking TODO исчез именно выбранный manager-корень; посторонняя или частичная уборка не может изобразить прогресс;
- до добавления scaffold агентскому протоколу нужен измеренный `NO_PROGRESS` для того же manager-candidate; повторная обычная remediation того же корня после `NO_PROGRESS` отклоняется CLI;
- каскадный `UNRESOLVED_SYMBOL` сам по себе не scaffold-eligible: сначала должен быть выделен конкретный project helper/POM-корень; его payoff включает зависимые каскадные TODO;
- assertions, API Selenium/Playwright, селекторы, waits и произвольные выражения запрещены;
- suppression и scaffolding не могут владеть одним методом или шаблоном;
- catch-all и wildcard в owner отклоняются валидатором;
- количество scaffold-корней и полностью scaffold-зависимых тестов ограничено на стандартный запуск;
- при наличии scaffold нельзя заявлять runtime-ready acceptance; результат маркируется как structural-only;
- `runtimeReady` при этом всегда остаётся `false`.

Целевой результат — `MIGRATED_WITH_SCAFFOLDING`: массовая структура тестов переписана и компиляционно связана, а небольшой честный список runtime-blocker-ов остаётся для изолированной доработки.
